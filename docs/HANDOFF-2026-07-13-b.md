## 0. 一行要約

**「1本の木だけを見る」システムから、「現地（周辺木・環境・経緯）の中でその木を見る」システムへ。**
近接木・現地環境・工事履歴・経年変化を決定論（JS）で整理し、Gemini には「推論」だけをさせる。
ただし **周辺情報に振り回されてアンカリングしない**ことを最優先の制約とする。

---

## 1. 背景と課題

現状の `buildSystemInstruction(record, config, query, stage)` は **単体の record 1本だけ**を AI に渡している。
一方で PERSONA には既に「周囲の下層植生や他個体の状態も重視する」「大木と下層植生の両方に異常があるなら
共通・環境要因を疑う」とある。**AI に「他個体を見よ」と命じているのに他個体データを渡していない＝ループが
閉じていない。** 本改修はこのループを閉じる。

### 樹木医学上の位置づけ
- 「木を見て、森を見る」＝個体は環境・周辺個体との関係の中でしか正しく評価できない。
- 衰退の螺旋モデル（素因＋誘因＋主因の複合）＝単一原因に決めつけない。
- 見えない部分（地下部・過去）は、**周辺と履歴からの推論**でしか近づけない。

## 2. 設計思想（★最重要・アンカリング回避）

本機能は **「腐朽菌の伝播を検出する機能」ではない**。実装中に腐朽菌へ寄せないこと。

### 3原則
1. **近接木は「感染源」ではなく「対照群」**。対象木を相対評価し、共通要因（環境）か個体固有要因かを鑑別するために使う。
2. **要因カテゴリに専用配線を作らない**。腐朽菌だけの判定リスト・専用フラグを作らない。虫害・生理的衰退・環境・
   人為・病害は等価。近接木の所見は**生のまま（ラベル変換せず）**渡し、解釈は AI と樹木医に委ねる。
3. **陰性情報も情報**。「近接木が健全」は空データではなく、**個体固有要因を示唆する積極的な情報**。

### アンカリングが起きる罠（実装時に踏まないこと）
| 罠 | やってはいけない実装 |
|---|---|
| ① 決定論フラグ | 「土壌伝染性菌フラグ」など特定要因の専用判定を作る |
| ② RAG検索クエリ汚染 | 近接木の菌名（ナラタケ等）を検索クエリに足す |
| ③ 知識ベース汚染（★今回追加） | 近接木由来で引いたRAGチャンクを、対象木の `--- 知識ベース ---` に混ぜて“対象木の証拠”に見せる |

### ★アンカリング対策4点（レビューで合意・本改修の肝）
本機能で「周辺に振り回される」を防ぐため、下記4点を**必ず**入れる。詳細は §6。

- **(A) パス2 RAG の分離**：近接木・現地由来で引いたチャンクは知識ベースに混ぜず、
  **別ブロック「参考資料（現地・近接木由来／対象木の直接証拠ではない）」**として明示する。
- **(B) 非対称原則**：*健全な近接木は候補を絞ってよい*（陰性は強い証拠）。*異常な近接木は鑑別候補を
  「増やす」だけで、他候補を消したり結論を上書きしたりしない*（陽性は弱い示唆）。
- **(C) 確度の結合**：結論が対象木自身の直接所見・実測ではなく現地コンテキストに依拠する場合、
  確度は必ず〔中〜低〕・「仮説」と明示。近接木の所見だけで確度を上げない。
- **(D) 機械判定を覆す許可**：`comparePattern` の P1/P2 は grade の粗い比較にすぎない。
  現物所見が判定と食い違う場合は現物を優先し、その旨を述べてよい。

## 3. コーパス外知見の扱い（★出典捏造の防止）

RAG コーパスは **『樹木医の手引き4版』『緑化樹木腐朽病害ハンドブック』の2冊のみ**。

- 「周囲の同樹種と比較して判断」「WGS84座標」「植栽間隔6〜8m」等は**『街路樹診断マニュアル』由来でコーパス外**。
  → **AI にこれらを出典【n】として引用させない**（捏造になる）。JS 側の決定論ルール／PERSONA 規則として実装し、
  出力上は **【システム規則】** と明示して書籍引用【n】と区別する（既存 t/R 比の決定論計算と同じ扱い）。

### 参考：コーパス内で裏が取れている記述（AI が引用してよい）
- ナラタケは土壌中に根状菌糸束を形成し、根が接触していなくても隣接木に感染しうる。
- シマサルノコシカケ（南根腐病）・キンイロアナタケ・マツノネクチタケ等は根の接触部から隣接木に感染。
- ベッコウタケも根系を介して隣接木に感染すると考えられている。
- 被害木の周囲に溝を掘る／プラスチック板を埋設して物理的に遮蔽する。
- **歯止め**：「根株腐朽菌の**すべてが**根系を介して感染するわけではない」（腐朽ハンドブック・明記）。
- 道路工事等による根の切断は、根株腐朽病害の発生を**誘発しやすい**（腐朽ハンドブック）
  → 「掘削 **or** 腐朽菌」の二者択一ではなく、**誘因（掘削・傷口）→ 主因（腐朽菌の侵入）の連鎖**として扱える根拠。

## 4. データモデル変更（★2階層：案件 → 現地 → 樹木）

### 4-0. なぜ2階層か（treeNo 衝突の回避）
`treeNo`（個体番号）は**案件が変われば重複**する（例「A公園診断業務」の No.1〜3 と「A市道診断業務」の No.1〜3）。
番号や座標だけで近接木を選ぶと、公園No.1 を診ているのに道路No.2 を拾う事故が起きる。

**近接木を混ぜない単位は「業務（案件）」ではなく「現地（物理的に隣接しうる樹木の集合）」である。**
1業務が複数の現地にまたがる場合（市全域の街路樹診断など）、同一業務内でも 北通りNo.1 と 南通りNo.1 は別物。
→ 3層に分け、**近接木・履歴の抽出は必ず「現地(site.id)」でスコープする**。案件は管理上のまとめに徹する。

| 概念 | 役割 | 近接木スコープに使うか |
|---|---|---|
| **案件（業務）** 例「A市街路樹診断業務」 | 一覧・レポート・GAS同期の括り、フィルタ | **使わない**（管理用） |
| **現地（場所）** 例「北通り 北側」「A公園」 | 隣接しうる樹木の集合。`treeNo` はこの中で一意 | **これで絞る（必須）** |
| **樹木（treeNo）** | 現地内の個体番号 | 現地スコープ内でのみ比較 |

### 4-1. site（現地）レコード（新設）
```js
{
  id: 'site_xxx',       // 主キー。record.projectId にこの id を格納
  name: '',             // 現地名（例: A公園 / 北通り 北側）
  projectName: '',      // ★案件名（業務名。例: A市街路樹診断業務）。管理・表示用。スコープには使わない
  kind: '',             // 公園 / 街路 / 社寺 / その他
  history: '',          // 経緯・履歴（工事・掘削・舗装改修・盛土・強剪定・薬剤・被害発生など）
  environment: '',      // 立地環境（土壌・排水・踏圧・植栽桝・日照・風・交通・塩害など）
  surroundings: '',     // 周辺情報（近隣でのナラ枯れ・カミキリ被害・枯損の発生情報など）
  notes: '',
  createdAt, updatedAt
}
```
> 新エンティティ「案件」は作らず、`site.projectName` の1フィールドで表現する（軽量・後方互換）。
> 将来、案件でのグルーピング一覧が要れば `projectName` で group-by すればよい（Phase B）。

`blankRecord()`（`src/App.jsx:17`）に **既に `projectId`/`treeNo`/`latitude`/`longitude` が定義済み**。
これらを起こして使う。**`record.projectId` に `site.id` を格納**する。
`treeNo` は「その現地の中で一意」がルール（現地が違えば重複してよい）。

## 5. 決定論エンジン（新規 `src/logic/siteContext.js`）

Gemini に距離計算や比較をさせない。**JS が計算し、結果だけを渡す。**
`evaluateRecord(record, enabledIds)` を再利用する際の `enabledIds` は、既存同様
`DECLINE_ITEM_IDS`（`declineItems.js`）の**モジュール定数**を import して渡す（`karteSummary.js:21` と同じ扱い。
ランタイム状態ではないので props で引き回さない）。

### 5-1. 近接木の抽出 `selectNeighbors(target, records, opts)`
1. **同一 `projectId`（現地）で絞る＝必須**。空なら近接木なし（＝P4）。← treeNo 衝突回避の要。
2. 緯度経度が両方にあれば Haversine で距離計算 → **半径 R（既定10m）以内**を距離昇順。
3. 座標が無ければ `treeNo` の**連番隣接（±1〜2本）**を近接とみなす。【システム規則・推定】
4. 上位 **最大 N 本（既定8）**。同一個体（同じ id）と自分自身は除外。
5. **同樹種を優先**して対照群にする（同樹種が2本未満なら全樹種で代替し、フラグに残す）。

opts は config から（`neighborRadiusM` 既定10 / `neighborMax` 既定8）。

### 5-2. 経年変化 `selectHistory(target, records)`
同一 `projectId` かつ **同一 `treeNo`（または同一 `nickname`）** の過去レコードを `createdAt` 昇順で最大3件。
★ガード：`treeNo`/`nickname` が**非空のときだけ**照合する（空だと同一現地の別個体まで誤って束ねる）。

### 5-3. 比較パターン判定 `comparePattern(target, neighbors)`
既存 `evaluateRecord()`（`diagnosis.js:52`）を再利用し、活力度平均・健全度最悪・総合判定で比較する。

**主パターン（排他）**
| ID | 条件 | AI に渡す示唆 |
|---|---|---|
| **P1** | 近接木の過半も不良（総合 C/D、または活力度平均が対象木と同水準） | 共通要因を優先鑑別（土壌・水・大気・気象・管理・工事など） |
| **P2** | 対象木のみ突出して不良、近接木の過半が健全（A/B） | 個体固有要因を優先鑑別（根の損傷・腐朽・傷・素因・局所的な人為） |
| **P4** | 近接木が2本未満（データ不足） | 判定保留。近接木情報が無いことを明示し、推論を膨らませない |

**併記フラグ（主パターンと同時に立ちうる）**
| ID | 条件 | 内容 |
|---|---|---|
| **P3** | 対象木と近接木で異常項目（健全度≠A／活力度評点≧2）が1つ以上一致 | 一致項目名を列挙。キノコを特別扱いしない（虫害・枝枯れ・葉の異常・傾き等すべて同じ） |

**出力**
```js
{ pattern:'P1'|'P2'|'P4', sharedFindings:['項目名', ...], sameSpeciesControl:true|false, neighborCount:n }
```

### 5-4. `buildSiteQuery(site, neighbors, history)`（パス2 RAG 用）
現地の `history`/`environment`/`surroundings` と、近接木の**異常項目ラベル＋所見テキストをそのまま**連結（上限300字）。
菌名リスト等のキュレーションは一切挟まない。`site` も `neighbors` も空なら空文字を返す（＝パス2スキップ）。

## 6. AI への注入（`src/logic/systemPrompt.js`）

### 6-0. 引数のまとめ方（★レビュー合意）
位置引数の肥大を避けるため、現地情報は **1オブジェクト `siteContext` に集約**して
`buildSystemInstruction(record, config, query, stage, siteContext)` の**末尾**に渡す。
```js
siteContext = { site, neighbors, history, pattern, sharedFindings, sameSpeciesControl }
```
`ChatPanel.jsx` の**2箇所**（`:86` チャット送信 / `:106` レポート生成）**両方**で同じ `siteContext` を渡す。
`siteContext` が未定義/空なら現行と完全に同じ挙動（後方互換）。

### 6-1. 現地コンテキストブロック（`--- 現在のカルテ ---` の後に追加）
```
--- 現地コンテキスト（対象木だけでは見えない情報）---
■ 案件: A市街路樹診断業務
■ 現地: 北通り 北側（街路）
■ 経緯・履歴: 3年前に電線共同溝工事で北側を掘削
■ 立地環境: 土壌固結、排水不良ぎみ、園路に接し踏圧あり
■ 周辺情報: 近隣路線でナラ枯れの発生報告あり
■ 経年変化（同一個体）: 2024-05 衰退度Ⅱ → 2026-07 衰退度Ⅲ
■ 近接木（同一現地・距離順／同樹種を対照群とする）:
  - No.2（8m｜ケヤキ）総合C・衰退度Ⅲ／根元にキノコ多数（ナラタケ）／幹に傷
  - No.3（12m｜ケヤキ）総合A・健全
  （他3本は健全）
■ 【システム規則】機械判定: P2（対象木のみ突出して不良／近接木の過半は健全）
   併記: P3（近接木と一致する異常項目: 根元の異常、幹の傷）
   → 個体固有要因を優先鑑別しつつ、一致所見の共通性も検討すること
```
- 近接木の所見は `fungusOther`/`findings` を**生のまま**。要因ラベルに変換しない。
- 健全な近接木も**必ず載せる**（陰性情報）。多い場合は「他◯本は健全」と1行集計。
- 近接木0本のときは省略せず **「■ 近接木: データなし（判定保留）」** と明示。
- `siteContext` 自体が無い（既存カルテで projectId 空）ときは、**このブロックごと省略**して現行の
  プロンプトと一致させる（“判定保留”の1行も出さない＝トークン増と余計な誘導を避ける）。

### 6-2. 2パス RAG（★アンカリング対策の核心 = §2-(A)）
`buildRetrievalQuery` に**菌名を足すのは禁止**。検索を2本立てにする。

| パス | クエリ | k |
|---|---|---|
| **パス1（主）** | 現行どおり：相談文＋対象木カルテ構造値（既存 `buildRetrievalQuery`、**変更しない**） | **8（現状維持）** |
| **パス2（従）** | 新規 `buildSiteQuery(site, neighbors, history)` | 4 |

- ★**パス1は k=8 を維持する**（初稿の k=6 は不採用）。`systemPrompt.js:83` の k=8 は「腐朽菌を列挙して」等の
  網羅質問の取りこぼし対策で、PERSONA「該当項目は漏れなく挙げる」と直結。ここを下げると網羅性が退行する。
- ★**パス2の結果は知識ベースに混ぜない**（§2-(A)）。`formatChunks` で1ブロックに混ぜると、対象木が腐朽所見ゼロでも
  隣のキノコで腐朽の章が対象木の知識ベースに入り込み、アンカリング源になる。
  → パス2チャンクは**別セクション**として出力する:
  ```
  --- 参考資料（現地・近接木の所見から引いた／対象木の直接証拠ではない）---
  （相談中の対象木そのものではなく、現地・近接木の状況から引いた背景資料。仮説の材料であり結論の根拠にしない）
  【S1｜書名 p.xx 〔章〕】本文…
  ```
- パス1・パス2それぞれ内部で重複排除（book + pageBatch + section）。パス1優先、合計上限は12に緩める
  （8＋最大4）。パス2が空/スキップ時は現行と同一。
- `refs` に **`origin:'対象木'|'現地'`** を持たせ UI で区別。プロンプト本文でも上記見出しで区別する。
- `site` も `neighbors` も空なら **パス2はスキップ**（＝現行と完全に同じ挙動）。

### 6-3. PERSONA 追記（末尾に追加。既存文は変更しない）
```
- 【現地コンテキストの原則】提示された「現地コンテキスト」は、対象木を相対評価するための対照群・背景情報である。次の順で用いる:
  ①現地全体・近接木にも同様の異常があるなら、共通要因（土壌・水環境・大気・気象・管理・工事等）を優先して鑑別する。
  ②対象木だけが突出して不良で近接木が健全なら、個体固有の要因（根の損傷・腐朽・傷・素因・局所的な人為）に絞り込む。近接木が健全であることは「情報が無い」のではなく、個体固有要因を示唆する積極的な情報として扱う。
  ③近接木に所見（キノコ・虫害・枝枯れ・葉の異常等）がある場合、それは「対象木にも同じことが起こりうる」という仮説の材料であり、対象木の主因が確定したことを意味しない。特に近接木にキノコがあっても、対象木の衰退主因が腐朽であると即断しない（資料に「根株腐朽菌のすべてが根系を介して感染するわけではない」と明記されている）。
  ④近接木・現地情報が無い、または少ない場合は「判定保留」とし、推論を膨らませない。乏しい情報から断定しないこと。
- 【非対称の原則】健全な近接木は「共通要因が無い」という強い陰性証拠なので、個体固有要因へ候補を絞る材料に使ってよい。一方、異常な近接木は弱い示唆にすぎないので、対象木の鑑別候補を「増やす」だけに用い、他候補を消したり主因を上書きしたりしない。周辺の派手な所見に引きずられて鑑別を1つに畳まないこと。
- 【確度の結合】結論が対象木自身の直接所見・実測ではなく現地コンテキスト（近接木・履歴・環境）に依拠している場合、確からしさは必ず〔中〜低〕とし「仮説」と明示する。近接木の所見だけで確度を上げない。
- 【機械判定の扱い】「【システム規則】機械判定(P1/P2/P3)」はグレードの粗い比較にすぎず、書籍の記述でもない。現物所見が判定と食い違う場合は現物を優先し、その旨を述べてよい。機械判定を絶対視しない。
- 【要因の等価扱い】現地コンテキストは腐朽菌の検出のためのものではない。腐朽菌・穿孔性害虫・生理的衰退・土壌/水/大気環境・人為（掘削・踏圧・盛土・強剪定・薬剤・機械傷）・気象を等価な候補として扱い、現地の実データが指す方向に従う。特定の要因に先回りして寄らない。
- 【過去の経緯】工事・掘削・舗装改修・強剪定等の履歴は、根の切断や傷口を通じて後年の異常の誘因になりうる。誘因（履歴・環境）と主因（病原・害虫等）を分けて記述し、連鎖として説明する（単一原因に決めつけない）。
- 【出典の区別】「【システム規則】機械判定」はこのシステムが計算した結果であり書籍の記述ではない。書籍引用は【n】で示し、機械判定・一般知識と混同させない。知識ベースに無い文献（街路樹診断マニュアル等）を出典として引用しない。
```

## 11. 動作確認シナリオ（Koh の実例）
案件「A公園診断業務」→現地「A公園」に登録し、No.1 でチャット開始。
- **site**: projectName=「A公園診断業務」, name=「A公園」, history=「3年前に電線共同溝工事で北側を掘削」
- **No.1**（対象木・ケヤキ）: 根元腐朽の所見あり／キノコなし／衰退度Ⅲ
- **No.2**（8m・ケヤキ）: fungusOther=「根元にナラタケ多数」／衰退度Ⅲ
- **No.3〜5**（ケヤキ）: 健全
- （別現地「B市道」にも No.1〜3 を登録しておき、A公園No.1 が B市道を拾わないことを確認）

**期待挙動**
- 現地ブロックに No.2 の所見が**生のまま**／No.3〜5 の健全も載る。
- 機械判定 P1/P2 を提示（＋P3 併記）。近接木は A公園内のみ（B市道は混ざらない）。
- 近接木由来の腐朽章は `--- 参考資料 ---` 側に入り、`--- 知識ベース ---` を汚さない。
- AI は「掘削 or 菌」の二択に落ちず、**誘因（掘削）→ 主因（感染の可能性）の連鎖**として提示し、
  **同時に踏圧・排水・桝の狭小等も鑑別に残す**。
- 「近接木にナラタケ→対象木も腐朽が主因」と**断定しない**（歯止め）。確度は〔中〜低〕・仮説明示。
- 出典【n】は腐朽ハンドブックの該当箇所を指し、**コーパス外文献を引用しない**。
