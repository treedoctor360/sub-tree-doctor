## 0. この道具は何を助けるのか

樹木診断で本当に難しいのは、**見えるものを読むこと**ではなく、
**見えない部分（地下部・過去の経緯・周辺環境）をいかに想像するか**である。
根の損傷も、何年も前の工事も、隣の木で起きていることも、対象木の外観には現れない。

サブ樹木医は、その「想像＝推論」を若手〜中堅の樹木医と**一緒に行う頭脳層**であり、
診断を代行する自動判定機ではない。数値判定や菌の同定は既存の道具に任せ、
**「所見と背景から、何を疑い、次に何を確かめるか」を対話で導く**ことに徹する。

### エコシステムの分担（重複を作らない）
- **shindanv27** … 診断項目の入力（健全度14・活力度17）と総合判定・所見・地図・Excel。
- **wood-decay-fungi** … 腐朽菌の同定（写真AI）。**種の確定はここに委ねる**。
- **サブ樹木医（本アプリ）** … 上記を受けて **2冊を根拠に推論し、鑑別・精密診断の指示・危険度の見立て・
  処置方針を導き、相談記録として残す**。空きを埋める設計で、他アプリの機能を再実装しない。

## 1. 診断の中核思想（5原則）

この5つは `src/logic/systemPrompt.js` の PERSONA に条文として実装されている。**崩さない**。

1. **複合要因で考える（主因・素因・誘因）**
   樹木の異常・衰退は単一原因とは限らない。素因（樹種・個体の弱さ）＋誘因（環境・立地・人為）＋主因が
   重なって起こる。**一つの原因に決めつけない。**（出典: 手引き4版「総合診断」「病気の概念」）

2. **腐朽に飛びつかない**
   子実体や腐朽の所見は「**腐朽が在る**」という事実であって、衰退の**主因とは限らない**
   （腐朽はストレスや損傷の“結果”＝二次的なことも多い）。まず他要因を広く鑑別してから、腐朽を位置づける。
   腐朽菌の“種の確定”は wood-decay-fungi に委ねる。

3. **見えない部分を推論する（現地コンテキスト）**
   対象木1本だけでは足りない。**周辺木・立地環境・工事履歴・経年変化**から仮説を立てる。
   「木を見て、森を見る」——個体は環境・周辺個体との関係の中でしか正しく評価できない。

4. **安全最優先だが、断定しない**
   危険度は **外観(VTA) → 力学(t/R比) → 区画(CODIT/防御層)** の順に見立てる。
   根株腐朽・大開口空洞＋傾きは、樹勢が良く見えても危険側で扱う。
   ただし**最終診断・伐採指示はしない**。判断材料と選択肢を整え、**最終判断と責任は樹木医**にある。

5. **確度を明示する**
   見立てには必ず〔確からしさ 高/中/低＋理由〕〔確定/未確定〕〔次に確かめる1点〕を添える。
   情報が乏しいほど確度は低とし、写真のみ・未計測では確度を上げない。**乏しい情報から断定しない。**

## 2. アンカリング回避 — このシステムが最も気を遣う設計

RAG＋LLM は「**引いた資料と与えた情報が結論を引っ張る**」構造を持つ。
だからこそ、便利さのために**思考を特定要因（とくに腐朽菌）へ寄せてしまう罠**を、
設計の随所で意図的に塞いでいる。ここを緩めると、道具が“もっともらしい誤診”を後押しする。

- **近接木は「感染源」ではなく「対照群」**
  周辺木のデータは、対象木を相対評価して「共通要因（環境）か／個体固有要因か」を鑑別するために使う。
  近接木の所見は**生のまま**渡し、要因ラベルに変換しない。

- **要因カテゴリに専用配線を作らない**
  「腐朽菌フラグ」のような特定要因だけの判定・リストを作らない。
  腐朽菌・穿孔性害虫・生理的衰退・環境・人為・気象は**等価**に扱い、現地の実データが指す方向に従う。

- **陰性情報も情報／非対称の原則**
  「近接木が健全」は空データではなく、**個体固有要因を示唆する積極的な情報**。
  健全な近接木は候補を**絞る**材料に使ってよい（強い陰性証拠）。一方、異常な近接木は弱い示唆にすぎず、
  鑑別候補を**増やすだけ**——他候補を消したり主因を上書きしたりしない。

- **RAG汚染を避ける2パス設計**
  検索クエリに**菌名を足さない**（クエリの偏り＝思考の偏り）。近接木・現地由来で引いた資料は、
  対象木の「知識ベース」に混ぜず、**「参考資料（対象木の直接証拠ではない）」として分離**して提示する。

- **出典の区別（捏造の防止）**
  システムが計算した結果は【システム規則】、書籍からの引用は【n】として区別させる。
  根拠となる知識ベース（必携2冊）に**無い文献を出典として引用させない**。

> 一言で言えば「**便利さのために思考を1つの答えへ畳ませない**」。広く疑い、確度で正直に語らせる。

## 3. 知識の扱い（RAG の境界）

- 根拠は **『樹木医の手引き4版』『緑化樹木腐朽病害ハンドブック』の2冊のみ**。
  範囲外・曖昧な点は、無理に答えず「**現地確認・実測が必要**」と正直に言う。
- 検索は決定論（コサイン類似度）で top-k を引くだけ。**網羅質問（例:「腐朽菌を列挙して」）の取りこぼしを
  減らすため再現率を重視**し、偏りをクエリに足さない。
- 精度上の“正常な非対称”を理解しておく: 埋め込みは DOCUMENT×QUERY が非対称なので、
  **self-match が 1.0 にならないのは正常**。コーパス生成側とクエリ側で**モデル/次元/taskType を必ず一致**させる
  （不一致はコサインを壊す）。

## 4. プライバシー・所有の思想

- **記録は端末内（IndexedDB）が一次**。共有スプレッドシート同期は任意の“おまけ”で、端末内が真実の源。
- **中継URL・APIキー・トークンは利用者が各自設定**する。公開ビルドに特定の GAS URL を埋め込まない
  （URL露出＝課金キーへの入口の露出）。
- **著作物本文（OCRテキスト・コーパス）はリポジトリ/Pages に置かない**。参照はするが、
  コード・コミット・出力へ本文を大量転記しない（短い要約・引用に留める）。

## 5. アーキテクチャがこの形である理由（簡潔）

作りの選択には、上の思想がそのまま反映されている。

- **既存資産を無改造で流用し、重複を作らない** … GAS中継（対話＋embed＋コーパス配信）・記録GAS・
  総合判定ロジックは shindanv27／既存リレーの流儀をそのまま踏襲。新しい仕組みを増やさない。
- **決定論は JS、推論だけを AI に** … 距離計算・近接木の抽出・比較パターン・総合判定は JS が計算し、
  **結果だけ**を AI に渡す。AI に計算や比較を任せない（再現性とアンカリング抑制のため）。
- **後方互換を壊さない** … 新機能は既存カルテを無効化しない。例: 現地コンテキストは未設定なら
  完全に現行と同じ挙動（＝ `siteContext` が無ければ従来のプロンプト）。DBマイグレーションは既存データを消さない。
- **GAS は手貼り運用** … リポジトリからデプロイしない。`gas/*.gs` を変えたらエディタへ貼り直し→
  プロパティ設定→**新バージョンで再デプロイ**（`/exec` URLは維持）。コード変更だけでは反映されない。

## 6. この思想に反する変更の例（レビュー時の赤信号）

- 特定の腐朽菌名・要因を、コードや検索クエリに**ハードコード**する。
- 近接木の所見から、対象木の**主因を断定**させる（「隣にキノコ→この木も腐朽が主因」）。
- 知識ベースに**無い文献**を出典【n】として引用させる／機械判定と書籍引用を混同させる。
- 確度の明示を省き、乏しい情報で**断定**させる。
- 既定のリレーURLの埋め込み、著作物本文のコミット、既存カルテを壊すマイグレーション。

これらに触れる変更は、便利さの外見に反して**道具の信頼性を損なう**。避けること。
